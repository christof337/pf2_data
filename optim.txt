## Observations documentées (améliorations futures, hors périmètre refactoring)

### 1. clean_text() — 3 variantes non fusionnables

Fichier : scripts/spells_mapper.py, scripts/ability_mapper.py
Modification voulue : unifier toutes les variantes de clean_text() dans utils.py
Pourquoi c'est mieux : cohérence, moins de duplication
Pourquoi ça casse (risque) : les trois implémentations ont des comportements différents —
  - monster_mapper / trait_mapper : collapse whitespace basique (migrée dans utils.py ✓)
  - spells_mapper : retire aussi les coupures de mots PDF (tiret + espace)
  - ability_mapper : préserve les sauts de ligne doubles, supprime les caractères °/\ufffd
  Fusionner sans valider le comportement sur les fixtures de sorts et de capacités
  risquerait une régression. À revoir quand les tests couvriront ability_mapper.

### 2. monster.xsd — richTextType n'autorise pas <spellRef>

Fichier : schema/monster.xsd
Observation : richTextType dans monster.xsd autorise <table>, <list>, <spell>, <scseecBloc>
mais PAS <spellRef>. Dans spell.xsd, richTextType n'autorise que <spellRef>.
Le tag <spell> dans monster.xsd (pour langSpecial par exemple) et <spellRef> dans spell.xsd
semblent remplir le même rôle sémantique avec des noms différents.
À harmoniser lors d'une passe de consolidation XSD (Phase 3).

### 3. monster.xsd — simpleListType mélange trait/language/immunity

Fichier : schema/monster.xsd, ligne 232
Observation : simpleListType accepte indistinctement <trait>, <language> et <immunity>.
Cette permissivité empêche la validation fine. Idéalement, chaque type de liste
devrait avoir son propre type XSD.
Non modifié : risque de casser la validation du dragon de référence.

### 4. ability_mapper.py — clean_text() préserve les sauts de ligne

Fichier : scripts/ability_mapper.py
Observation : la variante de clean_text() de ability_mapper préserve les sauts de ligne
doubles (re.sub(r'\n{3,}', '\n\n', text)) contrairement aux autres mappers qui
aplatissent tout. Ce comportement est intentionnel pour les descriptions multi-paragraphes
du glossaire. À documenter dans le code si ce mapper évolue.
